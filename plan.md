

# 会话总结：你想做什么（详细版）

## 1. 目标：给 Agent/LLM “外挂能力”，而不是做一个大一统 Proxy

你一开始的想法是做一个 **Claude/Anthropic API 兼容的 proxy**，让 Claude Code 之类的客户端能“以为在用 Claude”，但底层其实用开源模型；并且在 proxy 层拦截 image、web search 等能力缺口，用额外模型/服务补上。

后面你把方向收敛成更轻、更纯粹的路线：

> **不做 proxy，不替代模型**
> 而是做一套 “Agent Skill 外挂体系”，让任意 LLM/Agent 通过调用 CLI 的方式获得视觉、搜索、抓取等能力。

核心动机：这条路更通用、更低耦合、更容易演进，也更不容易被某个特定客户端协议（Claude Code / Anthropic API）绑死。

---

## 2. 核心原则：一个 skill 只做一件事（可组合），保持纯粹

你明确提出了一个非常关键的产品原则：

- **一个 skill 只干一个事情**
- 不做“全家桶”，不在一个 CLI 里塞多个职责
- 组合交给 Agent（或上层编排器），底层工具保持单一职责

因此能力拆成三个独立外挂：

1) **modlens**：只负责视觉
2) **modsearch**：只负责搜索
3) **modfetch**：只负责网页抓取（fetch/web fetch）

它们之间是“管道式组合”关系，而不是“一个工具内置所有流程”。

---

## 3. 你要解决的真实能力缺口

### 3.1 视觉外挂（modlens）
动机：很多开源模型/纯文本模型没有多模态，你要用外挂补齐：

- 输入：用户传入的图片（截图/文档/照片/图表等）
- 处理：OCR 或 caption（未来可加 VQA）
- 输出：结构化的“文本证据”
- 用法：上层 agent 把这个输出注入上下文，让纯文本 LLM “知道图片内容”

本质上你做的是：

> **image → text evidence 的适配层**
> 让“非多模态模型”也能消费图片信息（以文本的形式）。

### 3.2 搜索外挂（modsearch）
动机：很多 agent/模型没有稳定的 web search 能力（或客户端实现不同），外挂提供统一入口：

- 输入：query
- 输出：结果列表（title/url/snippet…）
- 用法：agent 先搜，再决定抓哪些链接

### 3.3 抓取外挂（modfetch）
动机：搜索只是拿链接，真正让模型“读网页内容”需要 fetch：

- 输入：URL
- 输出：页面正文文本（HTML 做可见文本提取；非 HTML 做 best-effort）
- 用法：agent 对搜索结果里的某些 URL 调用 modfetch，拿正文再总结/引用

---

## 4. 交互/调用体验：极简、扁平、给 Agent 用

你强调这些工具主要是 **给 agent 调用**，不是给人频繁手敲，所以你要的是：

- **不要二级子命令**（不要 `modlens extract ...`）
- 直接扁平参数（类似 `modlens -i ./img.png`）
- 命令短、参数短、可预测

这背后的目标是：

> **让 agent 调用成本最低、失败面最小**
> （少一层解析、少一层状态分支，减少工具错误类型）。

---

## 5. 分发策略：选择 “纯 JS/TS + npm 直接跑”，降低安装摩擦

你讨论了 Go vs JS/TS 的核心矛盾：**分发与安装摩擦**。

### 5.1 你否定/弱化 Go 的原因
你考虑过 Go 做 CLI，然后：
- 交叉编译各平台可执行文件
- 扔到 GitHub Releases
- agent 根据平台下载并执行

但你意识到这里有大量“摩擦点/坑”：
- 多平台产物维护成本（arch/os 组合）
- macOS 的执行授权/Gatekeeper/quarantine 体验
- Windows 的路径/权限/安全软件误报等
- “让用户装 Go 再编译”非常别扭（对普通用户/朋友局更是）

### 5.2 你最终选择 JS/TS + npm 的原因
你得出更符合实际的假设：

- 使用 Codex / Claude Code 这种 AI 编程工具的人，本地通常已经有 Node
- npm 分发可以让 agent 更容易自动安装/更新/运行
- 不需要处理跨平台二进制、权限、release 工厂

因此你拍板：

> **纯 JS/TS（npm 直接跑）**
> 让安装路径最短、成功率最高。

---

## 6. 你最终想要的整体效果（从 Agent 视角）

你的“外挂体系”最终应该让 agent 具备这种工作流：

1. 如果用户给图片 → 调 **modlens** 得到文本证据 → 注入上下文 → 再让 LLM 推理/总结
2. 如果用户问需要联网信息 → 调 **modsearch** 拿候选链接 → 挑选目标 → 调 **modfetch** 拉正文 → 让 LLM 总结并引用来源

一句话概括就是：

> **把“视觉 / 搜索 / 抓取”变成三个可组合的 CLI skill**
> 让任意 agent/LLM 通过工具调用获得这些能力，而不依赖模型原生多模态或内置 web 工具。

---

## 7. 命名与产品定位（你已经定下来的部分）

- 视觉外挂：**modlens**
- 搜索外挂：**modsearch**
- 抓取外挂：**modfetch**

命名风格统一：
- `mod*` 表示外挂/模块化能力补丁
- 后缀表示具体器官/能力（lens/search/fetch）

这套命名也天然支持你未来继续扩展更多“外挂技能”（比如 modpdf、modaudio、modcache…），但每个仍坚持单一职责。
